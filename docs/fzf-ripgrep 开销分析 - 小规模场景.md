# 小规模文本下的 fzf/ripgrep 开销分析

## 📊 核心问题

**如果只有 5k 行文本，fzf/ripgrep 的开销还大吗？**

---

## 📏 文本规模分类

| 规模 | 行数 | 字符数 | 文件大小 | 典型场景 |
|------|------|--------|---------|---------|
| **超小** | < 1k | < 50k | < 100KB | 单章节、README |
| **小** | 1k-10k | 50k-500k | 100KB-1MB | 小型文档、单本薄书 |
| **中** | 10k-50k | 500k-2M | 1-5MB | 标准技术书籍 |
| **大** | 50k-200k | 2M-10M | 5-20MB | 大型文档集合 |
| **超大** | > 200k | > 10M | > 20MB | 完整知识库 |

---

## 🧪 5k 行文本场景分析

## 📈 三种方案对比 (5k 行场景)

### 方案 A：全文搜索 (elasticlunr)

```
构建时:
  - 分词时间：2-3 秒
  - 索引大小：~800KB
  
运行时:
  - 加载索引：800KB
  - 内存占用：5-10MB
  - 搜索速度：< 50ms
```

```
  - 加载源文件：700KB

  - 搜索速度：50-200ms (取决于算法)
```

## 📊 详细对比 (5k 行场景)

### 内存占用对比

| 方案 | 基础开销 | 文本加载 | 索引/结构 | 总计 |
|------|---------|---------|----------|------|
| **全文搜索** | 5MB | 800KB | 800KB | **~7MB** |
| **fzf 风格** | 5MB | 700KB | 15-40MB | **20-45MB** |
| **ripgrep 风格** | 5MB | 700KB | 5-20MB | **10-25MB** |

**关键发现**:
- 即使只有 5k 行，fzf/ripgrep 的**内存开销仍然较大**
- 主要原因是**算法本身需要构建匹配结构**
- 但差距从 10-25x 缩小到 **2-5x**

---

### 搜索性能对比

| 方案 | 首次搜索 | 后续搜索 | 输入延迟 |
|------|---------|---------|---------|
| **全文搜索** | < 30ms | < 10ms | 无感知 |
| **fzf 风格** | 50-150ms | 30-100ms | 轻微感知 |
| **ripgrep 风格** | 20-80ms | 10-50ms | 轻微感知 |

**关键发现**:
- 5k 行下，性能差距**可接受**
- fzf/ripgrep 也能达到**亚秒级**响应
- 但全文搜索仍然**更快**

---

### 用户体验对比

| 特性 | 全文搜索 | fzf 风格 | ripgrep 风格 |
|------|---------|---------|-------------|
| **精确匹配** | ✅ 好 | ✅ 好 | ✅ 最好 |
| **模糊匹配** | ❌ 不支持 | ✅ 优秀 | ❌ 不支持 |
| **拼音支持** | ⚠️ 需额外实现 | ⚠️ 需额外实现 | ❌ 不支持 |
| **部分匹配** | ✅ 好 | ✅ 优秀 | ⚠️ 仅子串 |
| **相关性排序** | ✅ TF-IDF | ✅ 模糊得分 | ❌ 无 |

## 💡 5k 行场景的启示

### ✅ 小规模下 fzf/ripgrep **变得可行**

**原因**:
1. **文本本身很小**: 700KB 对现代浏览器不是问题
2. **内存差距缩小**: 从 25x 降到 3-5x
3. **性能可接受**: 50-200ms 用户可接受
4. **移动端可用**: 30-50MB 移动端可以承受

---

### ⚠️ 但仍有问题

#### 问题 1: 每次搜索都要扫描
```
全文搜索:
  - 预建索引 → 搜索 O(1)
  - 100 次搜索 = 100 × 10ms = 1 秒

fzf/ripgrep:
  - 每次扫描 → 搜索 O(n)
  - 100 次搜索 = 100 × 50ms = 5 秒
```

#### 问题 2: 无法离线搜索
```
全文搜索:
  - 索引加载后，完全离线

fzf/ripgrep:
  - 需要保持源文件可访问
  - 单页应用难以实现
```

#### 问题 3: 功能受限
```
全文搜索:
  - TF-IDF 相关性排序
  - 多字段搜索 (标题/正文)
  - 布尔运算 (AND/OR)

fzf/ripgrep:
  - 主要是文本匹配
  - 缺少高级搜索功能
```

---

## 🔧 混合方案：最佳实践

### 方案：小规模用 fzf，大规模用全文搜索

```javascript
// 伪代码
class HybridSearch {
    constructor(textSize) {
        if (textSize < 5000) {  // 5k 行阈值
            this.mode = 'fzf';
            this.data = loadRawText();
        } else {
            this.mode = 'fulltext';
            this.index = loadIndex();
        }
    }
    
    search(query) {
        if (this.mode === 'fzf') {
            return this.fuzzySearch(query);
        } else {
            return this.index.search(query);
        }
    }
}
```

**优势**:
- ✅ 小规模：享受 fzf 的模糊匹配
- ✅ 大规模：享受全文搜索的性能
- ✅ 自动切换，无需用户关心

## 💬 结论

### 对于 5k 行文本

**fzf/ripgrep 变得可行**，但仍有取舍：

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| **全文搜索** | 最快、最省内存 | 无模糊匹配 | ⭐⭐⭐⭐⭐ |
| **fzf 风格** | 模糊匹配、体验好 | 内存 3-5x | ⭐⭐⭐⭐ |
| **ripgrep 风格** | 精确匹配快 | 无模糊匹配 | ⭐⭐⭐ |

---

### 最佳实践建议

1. **< 5k 行**: 可以考虑 fzf 风格
   - 模糊匹配很有用
   - 内存开销可接受
   - 用户体验更好

2. **5k-20k 行**: 混合方案
   - 根据设备性能选择
   - 或让用户选择模式

3. **> 20k 行**: 全文搜索
   - 性能和内存更优
   - 用户体验一致
